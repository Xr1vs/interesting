Build options

Heads up on what this can and can't do: Roblox doesn't support shipping "encrypted" scripts that still run — whoever has Studio access to a place can always read a Script's Source there, full stop. The good news is that Server and Commands (the whitelist-check logic) run with RunContext = Server, which Roblox never sends to players or exploits in the first place — only client-side LocalScripts (the playback/sync script here) are ever exposed to someone running exploit tools. This option just strips comments/whitespace from those three scripts to make casual reading harder if you hand the file to someone else; it's minification, not real encryption. The Whitelist module is left fully readable either way, since you're meant to keep editing it here.

Build & Download

Takes the bundled XrAudioSystem.rbxm template, renames the show folder, writes every uploaded track from the Process tab into the show's Sound slots (up to 58 tracks), bakes in the whitelist above, and gives you back one updated .rbxm — insert it into Studio same as before, no manual editing.

Roblox Account

Checking whether this deployment has a server-side key configured…
Create a key at create.roblox.com/credentials with the Assets · Write permission for your user or group — and make sure Restrict by IP is set to Any IP, since Vercel/Railway don't use a fixed outbound IP (a locked-down key is the #1 cause of a 401 error here). It's stored only in this browser (localStorage) and sent only to this site's own upload endpoint, which relays it to Roblox — it never goes anywhere else. Uploaded audio still goes through Roblox's normal moderation review before it's usable.
Simpler option: instead of pasting the key here every time, set ROBLOX_API_KEY, ROBLOX_CREATOR_TYPE, and ROBLOX_CREATOR_ID as environment variables in your host's dashboard (Vercel → Project → Settings → Environment Variables; Railway → Variables tab) and leave these three fields blank. The server fills them in automatically — it's the official, ToS-compliant way to do this without touching the browser at all.

Lock Screen

The password is set in the site's source (CONFIG.passwordHash near the top of the script) as a SHA-256 hash, not the plain password — change it before you deploy. This is a simple front-door lock, not real access control; for anything sensitive, put the deployment behind your host's own auth (e.g. Vercel's password protection) too.
